Red-Black Trees are one of those data structures that everybody knows they *should* understand but most people never actually do. They live inside your operating system's process scheduler, your database's index engine (MySQL InnoDB, for one), and C++'s `std::map`. If you've ever wondered how a sorted collection stays balanced while still supporting O(log n) insert, delete, and lookup — this is the structure doing the work.

The problem with learning Red-Black Trees isn't the concept. It's the *mechanics*. A textbook description of rotations and recoloring rules is nearly useless without seeing them happen live. I've read those descriptions multiple times over the years and still felt uncertain whether I had actually internalized the algorithm or just memorized the rules.

So I built **rbtrees**: a React component that lets you insert, search, or delete a value and then step through every comparison, rotation, and recolor the algorithm performs — one frame at a time.

## What it does

The visualizer runs the actual Red-Black Tree algorithm and records every event as it happens:

- **Comparisons** — each node visited while traversing down the tree
- **Rotations** — left and right rotations that restore balance
- **Recolors** — the color flips that maintain the Red-Black invariants
- **Terminal states** — node inserted, value found, not found, or deleted

You can step forward and backward through this event log at your own pace. A floating indicator travels the search path and lands (on insert), hovers (on find), or fades (on not-found or delete). The active node gets a highlight ring at each step.

All the graphics are SVG — no canvas, no third-party chart library.

## Why step-by-step matters

Most tree visualizers animate the final state. You click "insert 42", the tree rearranges itself with a swoosh, and you're supposed to have learned something. But the learning happens in the middle steps — specifically in the moments where the algorithm detects a violation and decides between a rotation and a recolor.

Stepping through a violation gives you the ability to pause at the exact frame where the invariant breaks and ask: *why did it do that?* That's the question a fast animation can't let you ask.

## Installation

```bash
npm install rbtrees
```

```tsx
import { TreeVisualizer } from 'rbtrees'
import 'rbtrees/style.css'

export default function App() {
  return <TreeVisualizer />
}
```

`TreeVisualizer` is self-contained — no props, no external state. It manages its own tree and renders everything inline.

## Implementation notes

A few decisions worth documenting:

**React 18 + SVG only.** I wanted full control over the rendering without introducing a canvas abstraction or a chart library dependency. SVG gives you addressable elements with CSS transitions, which is exactly what you need for smooth node movement and color changes.

**Animation via CSS transitions, not JS timers.** Each step updates React state, which triggers re-renders. The transitions between positions and colors are handled entirely by CSS `transition` on the SVG elements. This keeps the animation logic out of the component and makes the step-by-step pause/play behavior trivial — you're just controlling when state updates happen.

**Recording events, not diffs.** The algorithm runs in full, recording a log of events, before any rendering happens. The UI then plays back this log. This separation means the algorithm code has no rendering concerns, and the rendering code has no algorithmic concerns.
